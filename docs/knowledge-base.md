# AI 视频生成知识库

## 调研记录

### [2026-03-08] AI 视频提示词六层框架
- **来源：** https://venice.ai/blog/the-complete-guide-to-ai-video-prompt-engineering
- **关键发现：** 高质量提示词由六层构成：主体、动作、环境、摄影、风格、情绪
- **详细内容：** 每层都应包含具体的、可视化的描述，避免抽象词汇。提示词应像"给摄影师的拍摄指令"一样具体。
- **对数据结构的影响：** 数据结构的模块划分应覆盖这六层，每层都要有足够的字段深度

### [2026-03-08] Reddit 万次视频生成经验总结
- **来源：** https://www.reddit.com/r/PromptEngineering/comments/1mvfcrr/
- **关键发现：** 六部分提示词结构（主体+动作+环境+摄影+风格+情绪），每次只描述一个核心动作
- **详细内容：**
  - 一个提示词只描述一个动作，复合动作会导致混乱
  - 导演/电影参考比模糊的风格词汇更有效（如"Wes Anderson style"比"whimsical"更精准）
  - 负面提示词对避免常见 AI 瑕疵至关重要
  - 镜头语言（景别+角度+运动）是影响视频质量的关键维度
- **对数据结构的影响：** 动作字段应强调"单一动作"原则；增加导演/电影参考字段；负面提示词需要预置分类

### [2026-03-08] Runway Gen-4 参数体系
- **来源：** https://help.runwayml.com/hc/en-us/articles/37327109429011
- **关键发现：** Gen-4 支持文生视频和图生视频，关键参数包括 duration(5/10秒)、resolution、aspect_ratio
- **详细内容：** Gen-4 强调"摄影指令式"提示词，支持摄像机运动控制、风格参考、角色一致性
- **对数据结构的影响：** 技术参数模块需要覆盖这些 API 参数

### [2026-03-08] 如何让 AI 视频更真实
- **来源：** https://inizioengage.com/insights/to-make-ai-video-look-real-we-have-to-make-it-look-less-perfect/
- **关键发现：** AI 视频的"假"感来自过于完美，引入不完美感（镜头瑕疵、手持抖动、曝光波动）能大幅提升真实感
- **详细内容：**
  - 真实的视频有镜头畸变、色差、暗角、眩光等光学瑕疵
  - 手持拍摄有自然的微抖动，不是完美稳定的
  - 曝光会随环境变化产生微小波动
  - 焦点不总是完美的，偶尔有轻微跑焦
- **对数据结构的影响：** 新增"镜头不完美感"字段组（B组），包括手持风格、镜头瑕疵、散景形状、曝光变化等

### [2026-03-08] 物理真实感的关键维度
- **来源：** 综合多篇调研
- **关键发现：** 消除"塑料感"需要描述表面材质、物理行为、微观元素
- **详细内容：**
  - 表面材质描述（皮肤毛孔、金属反射、织物纹理）让物体看起来真实
  - 物理行为（布料飘动、液体流动、烟雾扩散）增强动态真实感
  - 微观环境元素（灰尘粒子、雨滴、光斑）增加画面层次
  - 磨损/老化痕迹避免"全新感"
- **对数据结构的影响：** 新增"物理真实感"字段组（A组），包括表面材质、物理行为、微观元素、磨损痕迹等

## 经验教训

### [2026-03-08] 不应添加"渲染引擎"字段
- **问题：** 在视觉风格模块中添加了"渲染引擎"（Unreal Engine 5、Octane 等）作为独立字段
- **原因：** 混淆了 3D/游戏领域的概念和 AI 视频生成的概念。虽然提示词中写"Unreal Engine style"可能有微弱效果，但它本质上是一种视觉风格描述
- **结论：** 其他领域的术语如果要用，应作为"艺术风格"的枚举选项之一，而不是独立字段。新增字段前必须过"字段准入四问"

### [2026-03-08] 参数字段必须对应真实 API
- **问题：** 容易添加"看起来合理但模型 API 不支持"的伪参数
- **原因：** 从理论角度设计字段，而非从模型实际能力出发
- **结论：** 参数字段（非提示词字段）必须对应至少一个主流模型的 API 参数

## 提示词技巧

### 单一动作原则
- **适用场景：** 所有视频生成
- **具体做法：** 每段提示词只描述一个核心动作，不要写"他走过去然后坐下来再拿起杯子"
- **效果：** 避免 AI 模型在多个动作之间混乱，显著提升动作的流畅性和自然度

### 导演/电影参考优于抽象词汇
- **适用场景：** 需要特定视觉风格时
- **具体做法：** 写"in the style of Denis Villeneuve"而不是"cinematic and dramatic"
- **效果：** AI 模型对具体导演/电影名称的响应比抽象形容词更精准、更一致

### 摄影设备参考增强质感
- **适用场景：** 需要特定画面质感时
- **具体做法：** 写"shot on ARRI Alexa Mini LF with Cooke S7 lenses"或"shot on 16mm film"
- **效果：** 模型能模拟特定设备的画面特征（景深、色彩、颗粒感）

### 负面提示词分类使用
- **适用场景：** 所有视频生成
- **具体做法：** 按类别添加负面提示词：画质问题（blurry, low quality）、人体畸变（extra fingers）、AI 特征（plastic skin, uncanny valley）
- **效果：** 系统性地避免常见 AI 生成瑕疵

## 模型特性

### Runway Gen-4
- **支持的参数：** duration(5/10s), resolution, aspect_ratio, seed
- **提示词偏好：** 摄影指令式，支持摄像机运动关键词
- **已知限制：** 最长 10 秒，复杂多人场景容易出错

### Kling 3.0
- **支持的参数：** duration(5-10s), resolution(up to 1080p), aspect_ratio, motion_strength
- **提示词偏好：** 对中文提示词支持较好，运动控制精细
- **已知限制：** 极端运动场景可能出现变形

### Sora 2
- **支持的参数：** duration, resolution(up to 1080p), aspect_ratio
- **提示词偏好：** 对长描述理解力强，支持复杂场景
- **已知限制：** 物理规律理解有时不准确

### Veo 3.1
- **支持的参数：** duration, resolution(up to 4K), aspect_ratio, 原生音频
- **提示词偏好：** 对电影术语响应好
- **已知限制：** 需要 Google 生态

### Luma Ray 2
- **支持的参数：** duration, resolution, aspect_ratio, seed, guidance_scale
- **提示词偏好：** 对光影描述响应敏感
- **已知限制：** 人物面部一致性有时不稳定

### [2026-03-08] 五层提示词框架（VO3 AI）
- **来源：** https://www.vo3ai.com/blog/how-to-write-cinema-quality-ai-video-prompts-a-sora-2-and-veo3-prompting-masterc-2026-03-06
- **关键发现：** 高质量提示词由五层构成：镜头语言 → 光照氛围 → 动作运动 → 材质纹理 → 参考风格
- **详细内容：** 最佳提示词长度为 80-150 词（Sora 2）。超过 120 词常导致模型混淆。时间序列描述（"先...然后...当镜头拉远..."）是提升视频连贯性的关键技巧。提及真实相机名称和电影参考显著提升画面质感。
- **对数据结构的影响：** 验证了现有模块划分（摄影+光照+动作+材质+风格）的合理性。action 字段的 placeholder 应引导用户使用时间序列描述。

### [2026-03-08] Seedance 2.0 提示词结构
- **来源：** https://www.imagine.art/blogs/seedance-2-0-prompt-guide
- **关键发现：** 标准结构为 Subject → Action → Camera → Style，单个动作用现在时态清晰动词
- **详细内容：** 最佳长度 50-200 词。支持负面提示词。强调包含小环境运动（蒸汽升起、织物摇摆、反射波纹）增加真实感。
- **对数据结构的影响：** 验证了 micro_elements 字段的价值。

### [2026-03-08] LTX Studio 提示词指南
- **来源：** https://ltx.studio/blog/ai-video-prompt-guide
- **关键发现：** 提示词结构为 Subject → Action → Setting → Camera Movement → Lighting → Style
- **详细内容：** 强调渐进式细节方法（先基础后细化）。避免冲突指令、过于复杂的序列、模糊描述词。不要忽略运动信息和时间考量。
- **对数据结构的影响：** 验证了现有模块结构的合理性。

## 经验教训

### [2026-03-08] 过度拆分字段降低灵活性
- **问题：** 将"动作"拆为 action + action_physical_detail + interaction 三个字段，将"氛围"拆为 sensory_layers + time_progression + spatial_depth_layers 三个字段
- **原因：** 追求"看起来全面"，但 AI 模型只接收一段连续的自然语言提示词，不会区分这些子字段
- **结论：** 如果 AI 模型只能理解一句自然语言描述，拆成多字段反而降低灵活性。应合并为一个字段，通过 placeholder 引导用户覆盖各个方面

### [2026-03-08] 多镜头概念不属于单次生成
- **问题：** 添加了 previous_shot_description、continuity_elements、style_consistency、character_consistency 等字段
- **原因：** 混淆了"多镜头项目管理"和"单次视频生成"的边界
- **结论：** 当前主流模型不支持跨镜头上下文，这些字段的值无法写进提示词，应删除

### [2026-03-08] 后期操作不属于提示词
- **问题：** transition_in/transition_out（过渡效果）出现在运动模块中
- **原因：** 混淆了"AI 视频生成"和"视频剪辑"的边界
- **结论：** 淡入淡出、溶解、划变等是后期剪辑操作，AI 模型无法通过提示词控制

### [2026-03-08] 3D 渲染术语不等于 AI 视频术语
- **问题：** light_falloff（光线衰减）、caustics（焦散）、subsurface_scattering（次表面散射）使用了 3D 渲染引擎的专业术语
- **原因：** 从 3D 渲染领域借用概念，但 AI 视频模型不是渲染引擎
- **结论：** 如果效果可以用自然语言描述（如"阳光穿透耳朵边缘的红色透光"），应在场景描述中表达，而非作为独立的技术字段

### [2026-03-08] 2026年主流模型能力对比（fal.ai 综合评测）
- **来源：** https://fal.ai/learn/tools/ai-video-generators
- **关键发现：** 2026年主流模型新增三大关键能力：首尾帧控制、原生音频、运动迁移
- **详细内容：**
  - **Veo 3.1**：4K输出、首尾帧控制（first-last-frame-to-video）、参考角色一致性（reference-to-video）、原生音频
  - **Sora 2 Pro**：最长25秒、原生音频+对话口型同步、视频风格迁移（video-to-video remix）
  - **Kling 3 Pro (O3)**：运动迁移（motion-control）、原生音频+voice ID、首尾帧控制、cfg_scale调节
  - **Seedance 1.5 Pro**：原生音频、首尾帧控制
  - **Hailuo 2.3 Pro**：首尾帧控制（start/end image）、1080p
  - **Wan 2.2**：开源、最低成本、最高720p
- **对数据结构的影响：**
  - 新增 `end_frame_url`（首尾帧控制，多模型支持）
  - 新增 `enable_audio`（原生音频开关，影响价格和输出）
  - 新增 `motion_reference_url`（运动迁移，Kling 3 支持）
  - `input_image_url` 重命名为 `start_frame_url`（语义更清晰）

## 经验教训

### [2026-03-08] 数据项优化应跟踪模型能力演进
- **问题：** 迭代1只关注了字段精简（删除冗余），忽略了模型能力演进带来的新字段需求
- **原因：** 只做了"减法"没做"加法"，没有调研最新模型的 API 参数
- **结论：** 每轮迭代应同时做加法和减法——删除模型不支持的字段，新增模型新支持的字段


### [2026-03-08] 空间提示词与画面布局（迭代4）

**调研来源：** https://help.scenario.com/en/articles/spatial-prompting-for-videos-generation/

**关键发现：**
1. **空间提示词（Spatial Prompting）** — Veo 3 支持在首帧图像上绘制箭头、标注文字来引导运动方向和变化。这是一种新的输入模式，通过 start_frame_url 传入带标注的图像即可，不需要新增字段。
2. **画面布局描述** — 构图类型（framing）只描述规则（三分法、中心构图等），但缺少对画面中元素具体空间位置的描述。

**变更：**
- 新增 `composition_detail`（textarea）到 cinematography 模块
- 更新 `start_frame_url` 的 desc 提及空间提示词能力

**经验：** 新能力不一定需要新字段——空间提示词是一种使用方式，通过现有的 start_frame_url 即可支持。

### [2026-03-08] 各模型提示词"方言"差异（迭代5）

**调研来源：**
- https://medium.com/@creativeaininja/how-to-actually-control-next-gen-video-ai-...
- https://blog.fal.ai/kling-3-0-prompting-guide/

**关键发现：**
1. **Runway Gen 4.5** — "动力学雕塑师"哲学，需要描述**力和物理反应**（力-反应语法：重量词+交互动词+力学语言）
2. **Sora 2** — "物理模拟器"哲学，需要描述**因果链**（解释 WHY 而非 WHAT）
3. **Kling 3.0** — "视听编排师"哲学，需要**剧本式叙事**，支持结构化角色标签 `[Character A: 名称, 语气]: "台词"`
4. **Veo 3.1** — "渲染引擎"哲学，支持结构化数据+参考图输入

**变更：**
- 新增 `physics_forces`（力学交互描述）到场景描述模块 — 让用户描述"什么力作用在什么上"，Runway 和 Sora 对此响应极佳
- 新增 `physics_constraints`（刚性约束）到场景描述模块 — 明确指定不应变形的物体，解决 AI 生成中常见的物体形变问题
- 新增 `dialogue_characters`（角色声音定义）到音频模块 — 支持 Kling 3 的多角色对话标签系统
- 升级 `dialogue` 的引导 — 从纯台词升级为"动作+对话"交织的剧本格式

**经验：** 不同模型对同一场景需要完全不同的提示词"方言"。数据结构应该收集足够丰富的信息，让下游的提示词组装器能根据目标模型选择合适的组装策略。
