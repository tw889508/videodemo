# 开发规范

## 项目目标

设计一个**完美的结构化数据结构**，通过这个数据结构可以**组装出完美的提示词和参数**，驱动 AI 视频生成模型达到**最佳生成效果**。

编辑器只是数据结构的承载工具，数据结构本身才是核心产物。

## 核心原则

**只优化数据项，让其更全面、更细致，聚焦如何提高视频生成质量，不要管其他的。**

所有开发工作必须围绕这一原则展开：数据结构的每一个字段、每一个枚举选项、每一条描述引导，都应服务于"提高 AI 视频生成质量"这一唯一目标。不做无关的 UI 美化、不做多余的功能扩展，专注于内容本身的完整性和精准性。

## 数据结构的唯一使命

这套数据结构存在的唯一理由是：**填完之后，能拼出一段让 AI 视频模型生成出最佳效果的提示词和参数。**

每个字段必须回答一个问题：**它能让生成的视频变得更好吗？**

- 能 → 保留，并打磨到最细致
- 不能 → 删除，无论它看起来多专业

## 领域边界：一切围绕 AI 视频生成

本项目的唯一领域是 **AI 视频生成**（即通过文本提示词 + 参数驱动 AI 模型生成视频）。

**不是**游戏开发，**不是** 3D 建模，**不是**传统影视后期，**不是**视频剪辑，**不是**视频编码。

### 字段准入四问（新增或修改任何字段前必须回答）

1. **提示词相关性** — 这个字段的值会写进 AI 视频生成的提示词中吗？如果不会，不加。
2. **模型可理解性** — 当前主流 AI 视频模型（Runway、Kling、Sora、Veo、Luma、Hailuo、Wan 等）能理解并响应这个描述吗？如果模型无法理解，不加。
3. **效果可感知性** — 把这个字段的值写进提示词，生成的视频能产出**肉眼可感知的差异**吗？能 → 保留；不能 → 删除或合并。
4. **领域归属性** — 这个概念属于"提示词编写"或"模型参数"吗？如果属于下游工具链（剪辑软件、渲染引擎、编码器、播放器），不加。

### 常见跑偏陷阱

| 陷阱 | 表现 | 正确做法 |
|------|------|---------|
| **借用其他领域术语** | 添加"渲染引擎""编码格式""色彩空间"等字段 | 这些是 3D/后期/编码领域的概念，AI 视频模型不直接使用 |
| **混淆上游与下游** | 添加"字幕""水印""拼接""转场特效"等字段 | 这些是视频生成之后的剪辑/后期操作，不属于提示词 |
| **过度工程化** | 把一个简单描述拆成十个子字段 | 如果 AI 模型只能理解一句自然语言描述，拆成多字段反而降低灵活性 |
| **追求形式完整** | 为了"看起来全面"而添加模型无法响应的字段 | 每个字段必须有实际效果，宁缺毋滥 |
| **忽视提示词本质** | 添加的字段无法自然地融入一段英文提示词 | 最终产物是一段提示词文本 + 一组 API 参数，字段必须能映射到这两者之一 |

## 字段分类标准

数据结构中的字段分为两类，设计时必须明确区分：

### 第一类：提示词字段（Prompt Fields）

这些字段的值会被拼装成自然语言提示词文本，发送给 AI 模型。

**设计要求：**
- 字段值必须能自然地融入一段英文描述
- 枚举选项应使用 AI 模型能理解的术语（优先英文摄影/电影术语）
- 自由文本字段应提供具体的 placeholder 示例，引导用户写出模型能理解的描述
- 避免抽象概念，优先使用具象、可视化的描述

### 第二类：参数字段（Parameter Fields）

这些字段的值会作为 API 参数直接传给 AI 模型。

**设计要求：**
- 必须是主流模型 API 实际支持的参数（如 resolution、fps、duration、seed、guidance_scale）
- 数值范围必须符合模型的实际接受范围
- 不添加模型 API 不支持的"伪参数"

## 质量判断法

当不确定一个字段是否应该存在时，做以下测试：

> 写两段提示词：一段包含这个字段的信息，一段不包含。
> 用同一个 AI 视频模型生成两段视频。
> 如果两段视频有明显的、有意义的差异 → 这个字段有价值。
> 如果差异微乎其微或完全看不出 → 这个字段应该删除或合并到其他字段中。
