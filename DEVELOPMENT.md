# 开发规范

## 项目目标

设计一个**完美的结构化数据结构**，通过这个数据结构可以**组装出完美的提示词和参数**，驱动 AI 视频生成模型达到**最佳生成效果**。

编辑器只是数据结构的承载工具，数据结构本身才是核心产物。

## 核心原则

**只优化数据项，让其更全面、更细致，聚焦如何提高视频生成质量，不要管其他的。**

所有开发工作必须围绕这一原则展开：数据结构的每一个字段、每一个枚举选项、每一条描述引导，都应服务于"提高 AI 视频生成质量"这一唯一目标。不做无关的 UI 美化、不做多余的功能扩展，专注于内容本身的完整性和精准性。

## 优化重心：数据项本身优先于枚举选项

优化工作的核心是**数据项（字段）本身**，而不是枚举选项的增减。具体来说：

**数据项层面的工作（高优先级）：**

- 审查每个字段是否应该存在（增、删）
- 审查字段之间是否有重叠或缺失（合并、拆分）
- 审查字段的归属模块是否合理（移动）
- 审查字段的类型是否恰当（如文本 vs 选择、单选 vs 多选）
- 审查字段的引导信息（label、placeholder、desc）是否能帮助用户写出更好的提示词

**枚举层面的工作（低优先级）：**

- 枚举选项的增减是次要工作，只在数据项确定之后才进行
- 不要花大量时间打磨枚举选项的措辞，除非它直接影响提示词质量

**理由：** 一个多余的字段比一个缺失的枚举选项危害更大——多余字段会误导用户、增加认知负担、生成无效提示词；而枚举选项不全时，用户可以通过"自定义"输入来弥补。

## 语言规范：全部使用中文

本项目的所有面向用户的内容**全部使用中文**，不使用英文。包括但不限于：

- **枚举值（value）** — 全部中文，如 `"浅景深"` 而非 `"shallow"`
- **枚举标签（label）** — 全部中文，如 `"慢动作"` 而非 `"Slow Motion"`
- **枚举描述（desc）** — 全部中文
- **字段标签（label）** — 全部中文
- **占位提示（placeholder）** — 全部中文
- **字段说明（desc/tooltip）** — 全部中文
- **分组名称（group）** — 全部中文
- **模块名称和描述** — 全部中文
- **代码注释** — 全部中文
- **文档** — 全部中文

**唯一例外：** 代码层面的变量名、接口名、字段 key 使用英文（这是编程规范要求，不是面向用户的内容）。

**理由：** 本项目的用户是中文用户，数据结构的最终产物（提示词）的翻译/转换是下游环节的事，数据结构本身应以用户最易理解的语言呈现，降低使用门槛。

## 数据结构的唯一使命

这套数据结构存在的唯一理由是：**填完之后，能拼出一段让 AI 视频模型生成出最佳效果的提示词和参数。**

每个字段必须回答一个问题：**它能让生成的视频变得更好吗？**

- 能 → 保留，并打磨到最细致
- 不能 → 删除，无论它看起来多专业

## 领域边界：一切围绕 AI 视频生成

本项目的唯一领域是 **AI 视频生成**（即通过文本提示词 + 参数驱动 AI 模型生成视频）。

**不是**游戏开发，**不是** 3D 建模，**不是**传统影视后期，**不是**视频剪辑，**不是**视频编码。

### 字段准入四问（新增或修改任何字段前必须回答）

1. **提示词相关性** — 这个字段的值会写进 AI 视频生成的提示词中吗？如果不会，不加。
2. **模型可理解性** — 当前主流 AI 视频模型（Runway、Kling、Sora、Veo、Luma、Hailuo、Wan 等）能理解并响应这个描述吗？如果模型无法理解，不加。
3. **效果可感知性** — 把这个字段的值写进提示词，生成的视频能产出**肉眼可感知的差异**吗？能 → 保留；不能 → 删除或合并。
4. **领域归属性** — 这个概念属于"提示词编写"或"模型参数"吗？如果属于下游工具链（剪辑软件、渲染引擎、编码器、播放器），不加。

### 常见跑偏陷阱

| 陷阱 | 表现 | 正确做法 |
|------|------|---------|
| **借用其他领域术语** | 添加"渲染引擎""编码格式""色彩空间"等字段 | 这些是 3D/后期/编码领域的概念，AI 视频模型不直接使用 |
| **混淆上游与下游** | 添加"字幕""水印""拼接""转场特效"等字段 | 这些是视频生成之后的剪辑/后期操作，不属于提示词 |
| **过度工程化** | 把一个简单描述拆成十个子字段 | 如果 AI 模型只能理解一句自然语言描述，拆成多字段反而降低灵活性 |
| **追求形式完整** | 为了"看起来全面"而添加模型无法响应的字段 | 每个字段必须有实际效果，宁缺毋滥 |
| **忽视提示词本质** | 添加的字段无法自然地融入提示词 | 最终产物是一段提示词文本 + 一组 API 参数，字段必须能映射到这两者之一 |

## 字段分类标准

数据结构中的字段分为两类，设计时必须明确区分：

### 第一类：提示词字段（Prompt Fields）

这些字段的值会被拼装成自然语言提示词文本，发送给 AI 模型。

**设计要求：**
- 字段值使用中文，便于用户理解和填写
- 枚举选项使用中文术语，必要时可在括号中附注专业术语以辅助理解
- 自由文本字段应提供具体的中文 placeholder 示例，引导用户写出清晰的描述
- 避免抽象概念，优先使用具象、可视化的描述

### 第二类：参数字段（Parameter Fields）

这些字段的值会作为 API 参数直接传给 AI 模型。

**设计要求：**
- 必须是主流模型 API 实际支持的参数（如 resolution、fps、duration、seed、guidance_scale）
- 数值范围必须符合模型的实际接受范围
- 不添加模型 API 不支持的"伪参数"

## 质量判断法

当不确定一个字段是否应该存在时，做以下测试：

> 写两段提示词：一段包含这个字段的信息，一段不包含。
> 用同一个 AI 视频模型生成两段视频。
> 如果两段视频有明显的、有意义的差异 → 这个字段有价值。
> 如果差异微乎其微或完全看不出 → 这个字段应该删除或合并到其他字段中。

## 知识沉淀与复用

本项目的数据结构质量取决于对 AI 视频生成领域的理解深度。每次调研、实验、踩坑获得的经验都是宝贵资产，必须沉淀下来，避免重复犯错、重复调研。

### 强制流程

**每次开发前：**

1. 必须先阅读 `docs/knowledge-base.md`（知识库），回顾已有的经验和结论
2. 必须先阅读 `DEVELOPMENT.md`（本文件），确认设计原则和边界约束
3. 基于已有知识判断当前任务，避免重复调研已有结论

**每次迭代后：**

1. 必须重新阅读 `DEVELOPMENT.md`（本文件），确认本次变更没有违反任何设计原则和边界约束
2. 如果发现违反，立即修正后再提交
3. 这一步不可跳过——随着迭代深入，容易不自觉地偏离原则，定期回顾是纠偏的唯一手段

**每次调研后：**

1. 将查阅的资料来源、关键发现、可执行的结论写入 `docs/knowledge-base.md`
2. 按固定格式记录，确保可检索、可追溯
3. 如果发现与已有结论矛盾的新信息，标注冲突并更新结论

**每次开发后：**

1. 如果踩了坑（如添加了不该加的字段、枚举值设计不合理），将教训写入知识库
2. 如果发现了提升视频质量的新技巧，将其写入知识库
3. 如果修改了数据结构，记录修改原因和效果

### 知识库格式（docs/knowledge-base.md）

知识库按以下结构组织：

```markdown
# AI 视频生成知识库

## 调研记录

### [日期] 调研主题
- **来源：** URL 或资料名称
- **关键发现：** 一句话概括
- **详细内容：** 具体的有用信息
- **对数据结构的影响：** 应该新增/修改/删除什么字段

## 经验教训

### [日期] 教训标题
- **问题：** 发生了什么
- **原因：** 为什么会出错
- **结论：** 以后怎么避免

## 提示词技巧

### 技巧标题
- **适用场景：** 什么时候用
- **具体做法：** 怎么写
- **效果：** 对生成质量的影响

## 模型特性

### 模型名称
- **支持的参数：** 列表
- **提示词偏好：** 该模型对什么描述响应最好
- **已知限制：** 该模型做不好什么
```

### 知识库的使用原则

1. **先查后研** — 动手调研前，先在知识库中搜索是否已有相关结论
2. **只记有用的** — 不记流水账，只记对数据结构设计有实际指导意义的信息
3. **结论优先** — 每条记录必须有明确的结论或行动建议，不记纯粹的信息摘抄
4. **持续更新** — 知识库是活文档，旧结论被新发现推翻时必须更新

## 版本控制：每次迭代后提交远程 Git

每完成一轮有意义的迭代（数据项的增删改、知识库更新等），必须将变更提交到远程 Git 仓库。

**强制流程：**

1. 完成一轮迭代的代码修改
2. 确认应用能正常运行（无编译错误、无运行时崩溃）
3. 使用清晰的中文 commit message 描述本次变更的核心内容
4. 推送到远程仓库（`git push`）

**commit message 规范：**

- 格式：`[模块] 变更类型：简要描述`
- 示例：`[场景描述] 删除：移除 previous_shot_description 等多镜头相关字段`
- 示例：`[摄影控制] 合并：将 camera_shake 合并入 handheld_style`
- 示例：`[知识库] 更新：记录五层框架调研结论`

**理由：** 每次迭代的变更都应可追溯、可回滚。如果某次修改导致数据结构退化，可以快速定位和恢复。
